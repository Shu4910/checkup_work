# 社員活動モニタリングアプリケーション 仕様書

**バージョン**: 1.0（確定版）
**作成日**: 2026-02-22
**対象環境**: Windows 10 / 11

---

## 1. 目的・背景

社員が業務時間中にどのようなアプリケーション・Webサイトを利用しているかを記録し、
生産性の可視化・改善に役立てる。

---

## 2. 対象ユーザー

| 役割 | 説明 |
|------|------|
| **管理者** | ログをCSVで閲覧・分析する管理職・IT担当者 |
| **対象社員** | モニタリングされる社員（Windowsユーザー） |

---

## 3. 機能要件

### 3.1 アクティビティ記録（メイン機能）

社員のPCで以下の情報をバックグラウンドで収集する。

| 収集項目 | 詳細 |
|----------|------|
| **タイムスタンプ** | 記録日時（YYYY-MM-DD HH:MM:SS） |
| **マシン名** | コンピュータ名 |
| **ユーザー名** | Windowsログインユーザー名 |
| **アクティブアプリ名** | フォアグラウンドのプロセス名（例: `chrome.exe`, `excel.exe`） |
| **ウィンドウタイトル** | アクティブウィンドウのタイトルバー文字列 |
| **ブラウザURL** | ブラウザがアクティブな場合、現在表示中のURL |
| **ページタイトル** | ブラウザがアクティブな場合、HTMLの `<title>` |
| **状態** | `active`（通常） / `locked`（ロック中・離席） |

#### 対応ブラウザ（URL・ページタイトル取得対象）

- Google Chrome
- Microsoft Edge
- Mozilla Firefox

### 3.2 記録タイミング

- **方式**: 定期ポーリング
- **間隔**: 60秒ごと
- **スリープ・ロック時の挙動**:

| 状態 | 挙動 |
|------|------|
| PC ロック中 | ポーリングは継続し、状態を `locked` として記録する |
| PC スリープ中 | Windows のスリープイベントを検知してポーリングを**停止**する |
| PC スリープ復帰 | 復帰イベントを検知してポーリングを**自動再開**する |

> **バッテリー消費への配慮**（詳細は §6 参照）

### 3.3 ログの保存

- 保存形式: **CSV**
- 保存先: ローカルPC（パスは設定ファイルで変更可能）
  - デフォルト: `C:\ProgramData\ActivityMonitor\logs\`
- ファイル名: `activity_log_YYYY-MM-DD.csv`（1日1ファイル）
- 保存期間: **90日**（古いファイルは自動削除）

### 3.4 ログフォーマット

```csv
timestamp,machine_name,user_name,process_name,window_title,url,page_title,status
2026-02-22 09:01:00,PC-001,tanaka,chrome.exe,Google Chrome,https://mail.google.com/,受信トレイ - Gmail,active
2026-02-22 09:02:00,PC-001,tanaka,excel.exe,売上管理表.xlsx - Microsoft Excel,,,active
2026-02-22 09:03:00,PC-001,tanaka,,,,,locked
```

| フィールド | 説明 |
|-----------|------|
| `timestamp` | 記録日時 |
| `machine_name` | コンピュータ名 |
| `user_name` | Windowsログインユーザー名 |
| `process_name` | プロセス名（ロック中は空白） |
| `window_title` | ウィンドウタイトル（ロック中は空白） |
| `url` | URL（ブラウザ以外・ロック中は空白） |
| `page_title` | ページタイトル（ブラウザ以外・ロック中は空白） |
| `status` | `active` または `locked` |

### 3.5 常駐・自動起動

- PC起動時にWindowsサービスまたはタスクスケジューラで自動起動
- **システムトレイにアイコンを表示**する（社員への明示的な告知）
- アイコンの右クリックメニュー:
  - 「現在の状態を表示」（動作確認用）
  - 終了は管理者パスワードが必要（社員による勝手な停止を防止）

## 4. 非機能要件

| 項目 | 要件 |
|------|------|
| **パフォーマンス** | CPU使用率 5%以下・メモリ 50MB以下を目標 |
| **バッテリー消費** | スリープ中はポーリング完全停止。ファイルI/Oは1分に1回のみ |
| **対応OS** | Windows 10 (64bit) / Windows 11 |
| **インストール** | インストーラー形式（.exe）を提供 |
| **ネットワーク** | 不要（完全オフライン動作） |
| **セキュリティ** | ログ保存先フォルダへのアクセスを管理者権限に限定 |

## 5. 技術スタック（案）

| 用途 | 採用技術 |
|------|----------|
| 言語 | Python 3.12 |
| アクティブウィンドウ取得 | `pywin32`（Win32 API） |
| ブラウザURL取得 | `pywin32` / UIAutomation |
| スリープ検知 | `WM_POWERBROADCAST`（Win32 電源管理イベント） |
| ロック状態検知 | `WTS_SESSION_LOCK` イベント |
| データ保存 | CSV（標準ライブラリ `csv`） |
| 常駐化 | Windowsサービス（`pywin32`） |
| システムトレイ | `pystray` + `Pillow` |
| パッケージング | PyInstaller（単体 .exe） |

## 6. バッテリー消費への配慮（懸念点と対策）

ご指摘のバッテリー消費については、以下の設計で最小化します。

| 懸念点 | 対策 |
|--------|------|
| ポーリングによるCPU常時起動 | 60秒間隔 + スリープ時停止で影響を最小化 |
| ブラウザURL取得のコスト | UIAutomationはCPU負荷が一時的に上がる可能性あり。取得処理は1秒以内に完了するよう設計 |
| ファイルへの書き込み | 1分に1レコードのみの軽量な書き込み |
| スリープ中の動作 | `WM_POWERBROADCAST` でスリープを検知し、ポーリングスレッドを即時停止 |

> **結論**: 60秒ポーリング + スリープ時停止の構成であれば、バッテリーへの影響は**実用上ほぼ無視できるレベル**と想定されます。ただし、ブラウザURL取得（UIAutomation）が稀に重くなるケースがあるため、初期テスト時にCPU使用率を確認することを推奨します。

## 7. スコープ外（今回対象外）

- macOS / Linux 対応
- スクリーンショット機能
- Webダッシュボード・GUIビューア
- 集計レポート自動生成
- 複数PC間のログ自動収集・集約
- 特定サイト閲覧時のリアルタイムアラート
- 除外アプリ・URL設定

---

## 8. 法的・倫理的考慮事項

- 就業規則・雇用契約への明記、または社員への事前告知が推奨される
- システムトレイアイコンにより、社員はモニタリング中であることを認識できる
- 個人情報保護法の観点から、収集データの管理・アクセス制限を適切に行うこと
- 収集したデータを目的外利用しないこと
- 本アプリケーションの利用は、日本の労働関連法規に準拠して行うこと

---

## 9. 決定事項サマリー

| 項目 | 決定内容 |
|------|----------|
| 記録タイミング | 定期ポーリング（60秒間隔） |
| ログ閲覧方法 | CSVファイルのみ |
| 社員への告知 | システムトレイアイコン表示あり |
| ログ管理場所 | ローカル保存のみ |
| 対象PC台数 | まず1台で検証、将来的に複数台展開を想定した設計 |
| スクリーンショット | 不要 |
| ロック中の扱い | `locked` として記録継続 |
| スリープ中の扱い | ポーリング停止（バッテリー配慮） |
| ログ保存期間 | 90日（自動ローテーション） |
| 除外設定 | なし |
